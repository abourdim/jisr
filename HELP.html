◈ Getting Started

1. Start the server

./start.sh
# Opens at: http://localhost:7411

2. Index your project

  1. Enter the absolute path to your C source directory, or click Browse
  2. Click ◈ Index Project
  3. Watch the progress bar — large projects index in 5–30 seconds
Tip: Your last indexed path is remembered. Next visit auto-fills the field.

3. Explore

  1. The function list (left sidebar) shows all functions ranked by call count
  2. Click any function → trace mode shows its callers and callees
  3. Or type in the Trace function box with autocomplete
  4. Click any graph node to trace it
  5. Use ◀ ▶ or Alt+←/→ to navigate your trace history

4. Switch views

ƒ FunctionIndividual function call graph with depth control
⬡ ModuleDirectory-level architectural overview
⊞ ExpandFunctions grouped inside module bubbles
▦ MatrixModule coupling heatmap — click cell to filter

? FAQ

What C projects does this work with?

Any C project parseable by cflow. Best suited for embedded firmware, RTOS code, drivers, and C libraries. C++ files (.cpp) are ignored.

How long does indexing take?

About 1 second per 10 source files. A 70-file project indexes in ~5 seconds. Results are cached in the browser for 24 hours — reopening the page restores your session instantly without re-indexing.

My graph has 500+ nodes and is slow.

Use the Depth slider to limit the graph to 2–4 levels. Or use Trace mode on a specific function — trace mode only renders the subgraph around one function, always fast.

Some functions are missing.

cflow follows explicit function calls. Functions called only via function pointers, macros, or inline assembly may not appear. External library functions show as gray "external" nodes — hide them with Ext.

Can I save my graph?

Yes — three ways: (1) SVG / PNG export buttons save the current view, (2) the session auto-saves to localStorage and restores on next visit (24h TTL), (3) click ✕ Session in the topbar to clear.

What does Re-index vs + New do?

⟳ Re-indexRe-parses the same source directory (use after code changes)
＋ NewClears everything and lets you index a different project

Is my source code sent anywhere?

No. Everything runs locally. The server runs on your machine, the browser connects to localhost. No analytics, no telemetry, no external requests.

Can I run this on a remote server?

ssh -L 7411:localhost:7411 user@server
# Then open http://localhost:7411 in your local browser

▶ How-To Guides

Find dead code

  1. In Analysis, click ☠ Dead code
  2. View unreachable functions with their source files
  3. Click Show in graph → dead functions glow red, others dim
  4. Click Export list → saves .txt file
Auto-excluded entry points: main, ISR_*, irq_*, interrupt_*, handler_*, init_*, _init, startup (case-insensitive)

Analyse change impact

  1. In Analysis, click ⚡ Impact
  2. Enter changed function names (one per line or comma-separated)
  3. Click ⚡ Analyse impact
  4. See affected count per module
  5. Click Show in graph → red = changed, orange = affected callers

Find a call path between two functions

  1. In Analysis, click ⇢ Path find
  2. Enter From and To function names
  3. Click ⇢ Find paths
  4. Results appear ranked shortest-first — click any to render it as a step-numbered graph
Tip: If no path found, try swapping From/To. The functions may be in separate call trees.

Apply the complexity heatmap

  1. In Analysis, click 🌡 Heatmap
  2. Review the top 10 riskiest functions
  3. Click Apply to graph → nodes recolored green→yellow→red
  4. Click any function name to trace it immediately
  5. Click Clear to restore default colors

Compare two call trees

  1. In Intersection trace, enter Function A and Function B
  2. Click ⊕ Intersect
  3. Graph shows only shared callees: blue = A root, purple = B root, green = shared

Explore module architecture

  1. Click ⬡ Module
  2. Circle diagram: nodes = modules (directories), edge thickness = call volume
  3. Click a module node → jumps to Function view for that module
  4. Click ⊞ Expand → functions inside module bubbles, cross-module wires
  5. Click ▦ Matrix → hover cells for counts, click to filter

Export for documentation

  1. Set up the view you want (trace, heatmap, path, etc.)
  2. Click SVG for vector — scales perfectly in Word, Notion, Confluence
  3. Click PNG for raster — 2× resolution, dark background baked in

◈ Understanding the Graph

Node colors

ColorMeaning
Blue borderProject function — defined in your source
Dark gray borderExternal function — from library or system header
Gold borderMacro-defined function
White borderCurrently focused / traced function
Red borderCaller of the traced function
Green borderCallee of the traced function

Edge colors (trace mode)

ColorMeaning
Red →Calls going OUT from the traced function
Green →Calls coming IN to the traced function
Dark blue →Other edges in the graph

The number badge (top-right corner of node)

Shows in-degree — how many other functions call this one. Higher = more central to the codebase. Functions with in-degree 0 and no matching entry-point pattern are candidates for dead code.

Depth slider

∞ (0)Show all reachable functions from all roots
1Root functions + their direct children only
2–4Recommended for large projects (>100 functions)

⚗ Analysis Tools Reference

☠ Dead Code Detector

Algorithm: finds project functions where no project-level function calls them (in-degree from project = 0).

Auto-excluded: main, ISR_*, irq_*, interrupt_*, handler_*, init_*, _init, startup

Limitation: functions called only via function pointers appear dead even if they are live.

⚡ Change Impact Analysis

Algorithm: reverse BFS — starts at the changed functions and walks upward through all callers.

Output: total affected count, breakdown by module, graph overlay (red = changed, orange = affected).

🌡 Complexity Heatmap

Score formula: (out_degree × 2) + in_degree + call_depth

Out-degree is weighted 2× because fan-out drives complexity more than fan-in. Call depth is computed via BFS from all root functions.

Color scale: green (simple) → yellow (medium) → red (complex)

⇢ Call Path Finder

Algorithm: DFS with backtracking. Limits: max 20 paths, max depth 12 hops.

Output: paths sorted shortest-first, each rendered as a step-numbered graph on click.

⊕ Intersection Trace

Algorithm: full descendant BFS for both functions, then set intersection.

Output: shared subgraph — blue = root A, purple = root B, green = shared callees.
